# v1.5 对账起点对齐重放 + 回测覆盖诚实化 【已完成 2026-07-23】

## 背景(#3533 实测定案)

对账原口径拿"库里全量回测的成交"切实盘窗口来配对。两个缺陷同一天被 #3533/#3677 暴露:

1. **起点状态不对齐(主)**:全量回测是从数据起点连续模拟过来的,进入实盘窗口时可能
   **正持着仓**(单仓互斥 → 后续信号全被堵);实盘 runner 是空仓上线。#3533 实测
   8 笔错 6 笔全是这种假差异:回测 7/16 的 BUY 持到 7/21,把实盘 4 笔 SELL 全堵成
   "回测未触发";实盘持仓又把回测 7/21 的 BUY 堵成"疑似 runner 漏单"(冤枉)。
   两边同时空仓的时刻,信号分钟级一致 —— 证明信号层无病,是比对口径的病。
2. **to_time 谎报(次)**:backtests.to_time 存的是"请求截止"(默认=点火时的 now),
   不是"实际吃到的最后一根 bar"。数据滞后于 now 时,回测伪装成覆盖到 now,
   把"数据没到"的缺口错标成"真差异"(#3533 的 7/23 14:00 笔)。

## 改法(两处, 均已上)

1. **对账重放**(routes/backtests.py `compute_reconcile`):比对不再取库里全量成交切片,
   而是**从实盘上线时刻(窗口起点-容差)空仓起跑、用最新 M1 内存现算一遍**:
   - 引擎加 `start_ts` 参数(services/backtest.py):该时刻前只喂指标不开新仓;
     **默认 None = 与历史行为逐字节一致**(合成数据平行验证通过);
   - 热身回看 3 倍 + 2 天余量(周末/缺分钟安全);重放失败降级为"回测侧无数据",不挡页面;
   - 重放不落库;**排名/成绩/回测分析继续用全量回测**(backtests 表),两个口径各答各的:
     全量答"策略历史赚不赚",重放答"回测能不能复现实盘"(尺子);
   - 红利①:重放永远新鲜 → **"回测过期(bt_stale)"缺口类别消亡**,归因只剩
     真差异(not_triggered)/ 数据未下载(data_missing)两类,后者补数据即自愈,无需重跑回测;
   - 红利②:对账不再依赖"跑过回测" —— 没回测过的策略照样能对账。
2. **to_time 诚实化**(services/jobs.py):写库时 to_time 记"实际读到的最后一根 M1"
   (`cache["m1"]["time"][-1]`),不再用请求的 now。全量回测的覆盖显示从此可信。

web 模板同步:对账页说明标注重放口径;"全部 N"改"重放期内 N";数据落后提示
去掉"再重跑回测";删除两处死掉的 bt_stale 分支。

## 尺子语义(定版)

- 对账分只受三件事影响:**信号一致性 / 执行一致性 / 数据覆盖** —— 持仓状态错位、
  回测新旧这两类"与策略无关的假差异"已从分数里根除;
- 分数用法不变:先看对账分再看盈亏(demo 亏但对账高=回测可信策略真不行,放心淘汰;
  demo 赚但对账低=运气,更危险)。
