# v2.2 AI 自动分析 · 生成 · 调参(单策略闭环)

> 定位:AI 闭环第二件。吃 v2.1 成绩单 → AI 提议参数 → 生成子代 → 回测 → 家族对比 → 迭代。
> 原则:**每一步页面可视、可手动**;自动化只是把步骤间的"人肉搬运"接上,步骤本身不变。
> 页面:「分析 ▾ → AI 策略分析」。产物全程 CANDIDATE,进 demo/live 照走漏斗,AI 不开后门。

## 已实现(2026-07-17,手动三步 + 底座)

| 步 | 内容 | 要点 |
|---|---|---|
| 1 | ID → 成绩单 | 与「策略分析」页 **同一个** `/strategies/{id}/report`(单一来源,那边改这里自动跟);页内可展开看 JSON;每次「载入」实时重取,提示词跟着同步 |
| 2 | 生成提示词 | api 单一来源 `/strategies/{id}/ai_prompt`(页面/prompt.txt 同文)。**返回协议**:`{"template": "...", "combos": [{"params": {...}, "basis": "..."}]}`,无 id;三层加固:规则明说(首字符 `{`、禁围栏、标准JSON)+ 填好数字的示例 + 末尾重申 |
| 3 | 粘贴 AI 返回 → 解析预览 | 纯前端、零写入;容错(自动剥围栏/前后废话);核对 template ✓/✗;逐组表格:参数/依据/格式检查(键完整 + 范围越界红字) |

## 待办(打通顺序)

1. **第 4 步 · 生成实例**:第 3 步绿✓的组一键接 `ai_candidates` → 逐组结果表(新ID / 已存在#ID / 拒收原因)+ 回读核验(库内参数与请求逐字段一致);红字组自动丢弃
2. **第 5 步 · 回测**:新 ID 预填、手动点启动(`ai_backtest` → `/backtest/run` 按ID,点名不走交叉门槛)
3. **第 6 步 · 结论与迭代**:家族对比表挂步骤号;"用它继续 ▸" 以最优子代为新父拉最新成绩单进下一轮;淘汰者去排名页按死因码归档
4. **第 2↔3 步自动打通(可手动可继承)**:AI 桥 = `claude -p` 的 HTTP 包装(Team 订阅零 API key),地址走已有 `ai_generator_url` 配置;页面加"一键问AI"把提示词送出、JSON 自动填回第 3 步 — **设计已讨论定稿,用户确认时机后再做**;半自动脚本 `scripts/ai_tune.py`(拉 prompt.txt → claude -p → 提交)可先用
5. 迭代纪律(攒经验后固化进 config):每轮 8~12 组、最多改 2 维、留出段亏=死因 `holdout_loss` 归档、连续 N 轮无改善停止

## 协议(第 2/3/4 步的合同, 改动需三处同步评审)

```json
{"template": "ma_cross", "combos": [
  {"params": {"fast": 17, "slow": 30, "sl_points": 1350, "rr": 1.5}, "basis": "一句依据(引用成绩单数字)"}
]}
```
- combos 恰好 count 项;params 键 = 模板参数空间的键(缺/多都拒);值在范围内按步长对齐
- 校验三层:页面预检(第3步, 提示) → api 收货(第4步, 拒收) → 唯一约束(库, 兜底去重)
- **统一收货管道**(2026-07-17 重构):所有参数来源(生成页 随机/网格/旧AI协议 + AI调参 + 未来DSL)
  都生产本协议的 combos,统一走 `services/instances.create_instances()`(校验/入库/去重/parent_id/
  回读核验/逐组反馈)——改校验或插入行为只改这一处;basis 记录来源(random/grid/ai/依据句)
