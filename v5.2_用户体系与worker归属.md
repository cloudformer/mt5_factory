# v5.2 用户体系与 worker 归属(第二步 · 只加不变)

> 序列见 v5.1。核心模型(与 Frank 对齐):**建用户=插一行;配置层默认全空,
> 改了才有自己的行(不复制快照);worker 直接挂 owner 列**。

## 一、users 补认证关键信息 + api key

```sql
ALTER TABLE users ADD COLUMN password_hash TEXT;   -- web 登录(bcrypt); 空=只能 api key

CREATE TABLE api_keys (
    id           SERIAL PRIMARY KEY,
    user_id      INTEGER NOT NULL REFERENCES users(id),
    key_hash     TEXT NOT NULL UNIQUE,   -- 只存哈希, 明文仅签发瞬间显示一次
    name         VARCHAR(64),            -- 备注: 谁的哪把钥匙
    enabled      BOOLEAN NOT NULL DEFAULT TRUE,
    last_used_at TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

一人可多把 key(网页/脚本各一把,泄露吊销单把);本步**只建表+签发能力,不接认证**
(认证中间件属于行为变更,归 5.6)。

## 二、user_config 覆盖层(关联到 config)

```sql
CREATE TABLE user_config (
    user_id INTEGER NOT NULL REFERENCES users(id),
    key     VARCHAR(64) NOT NULL REFERENCES config(key),  -- 只能覆盖真实存在的全局键(约束执法)
    value   JSONB NOT NULL,
    PRIMARY KEY (user_id, key)     -- 一人一键一行, UPSERT 覆盖, 免维护
);
```

- **读取规则**:用户自己的行有 → 用;没有 → 落 config 全局默认;
- **不复制**:新用户配置层为空,没改过的键实时跟随全局(无快照漂移);恢复默认=删自己那行;
- 外键附送保护:有用户覆盖的全局键删不掉(与 032 防删触发器互补)。

## 三、worker 归属(设计决策: 直接挂列, 不用关联表)

```sql
ALTER TABLE mt5_hosts ADD COLUMN owner_id INTEGER NOT NULL DEFAULT 1 REFERENCES users(id);
```

**为什么不用关联表**:一台 worker 必须属于且只属于一个用户(1:N)——关联表是 M:N 语义,
暗示"worker 可共享",违背"worker 完全隔离";真金通道归属一列定死,查询最简。
账户归属经 host 传递(trades.account 随 host 上报),不加字段;账户唯一索引(002)照旧防双跑。
本步只打标——"A 的 worker 只拉 A 的策略"的执法在 5.6。

## 风险 / 难度 / 验收

- 风险:**无**(三张新表/新列,全默认值,无人读取即无行为);难度:低;
- 验收:全站回归无差异;建测试用户+发 key+写覆盖配置+删覆盖回默认,全流程 SQL 可走通。
