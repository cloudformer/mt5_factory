# v1.6 关2 对账(回测 vs 实盘逐笔)— 数据层设计

> 状态:**设计定稿(2026-07-15 讨论确认),待开工**。
> 目标:量化"回测信号/结果 与 实盘(demo/live)实际成交 对不对得上",作为**回测质量标准**的一维,并支撑关2门禁。
> 背景/纪律见 `v1.2_数据持久化.md §9`(三层对账 + 阈值模板)、`v1.3_回测质量.md`(回测质量标准)。
> 核心认知:**回测与 runner 跑同一份 `on_bar`,信号层理论上应 ~100% 一致;对不上 = 数据/对齐 bug,不是策略问题。**

---

## 1. 涉及的表:1 源 + 1 派生(+ 复用现有)

| 表 | 角色 | 状态 | 维护方式 |
|---|---|---|---|
| `backtests.trades`(JSONB) | 回测逐笔信号 | 已存在,复用 | 随回测覆盖 |
| **`trades`**(新) | 实盘/demo 逐笔成交(**源数据/事实**) | 本期建 | **只增 upsert,永久保留** |
| **`reconciliations`**(新) | 对账结果(**派生/结论**) | 本期建 | **覆盖式,可随时重算** |

源(事实)与派生(结论)分开 —— 决定了各自的维护模型(见 §5)。

## 2. `trades` 表(源数据)

**粒度:回合(round-trip)**,一次完整开平 = 一行。**只存已平仓**(已发生→回写;持仓中→仍实时拉 MT5 不进库,遵 v1.2 §2 边界)。

```
trades
  account        MT5登录号          ┐ 复合主键 (account, position_id)
  position_id    券商唯一仓位ID      ┘ ON CONFLICT DO NOTHING → 重复同步只留1行
  strategy_id    magic-100000(手动/测试=NULL)   归因
  magic          原始 magic
  env            DEMO | LIVE         ← 每笔如实记(事实); 对账暂不分, 未来按此拆
  symbol
  direction      buy | sell
  volume
  entry_time  entry_price           券商服务器时间
  exit_time   exit_price
  sl  tp                            下单时挂的服务端保护
  close_reason   sl | tp | manual
  profit  commission  swap          券商真实数
  net_points     换算成点(与回测 points 同口径, 便于比)
  created_at  updated_at
```

- **PF / 回撤不入本表**:有每笔 `profit` + 时间序,PF(毛利/毛损)、最大回撤都能从行集算出 → 属派生指标,算完存 `reconciliations.metrics`。本表只存原子事实。
- **落全部已平仓回合,不按 magic 过滤 → 与 MT5 100% 一致**(MT5 已平仓成交的持久化镜像)。`strategy_id`=magic-100000(策略区间)否则 NULL(手动/测试单也落,记 NULL)。**"只看策略单"的过滤放到对账/分析时**(`WHERE strategy_id IS NOT NULL`)—— 落库层不做业务过滤,保真。
- 心跳增量同步:数据源 = bridge `/trades`(现成),API 心跳循环 upsert。四道锁同 v1.2 §3。
- **强壮性(D1/D2)**:①逐笔容错 —— 单个回合坏数据(缺字段/畸形)只跳过它并记日志,不拖垮整批;②落库后一致性自检 —— 按 position_id 核对"本批 MT5 给的回合是否都进了库",缺了则 `warning` + 主机事件 `trades_mismatch`(把"与 MT5 一致"从口头保证升级为自动验证 = v1.2 §3 第四道锁)。
  **窗口自适应**:空库回填 90 天(足够大,历史完整);稳态只拉"最新平仓往前 3 天"(增量,靠主键去重,大而不浪费)。要纳入更早历史 → 临时调大回填天数跑一轮。
- 量级:少策略 × 低频 → 一年几千行/账户,忽略不计。

## 3. `reconciliations` 表(对账结果)

**一行 = 一个 (strategy_id, scope),覆盖式(latest verdict)。**
`scope` = 这次对账"回测 vs 哪批实际成交"的口径,**真实语义不是占位**:现阶段只算 `all`(程序把该策略 demo+live 成交合并与回测比);未来要分时,同一策略再算 `demo`/`live` 两行(各只 filter 对应 env),**只加行不改表**。

> 区分清楚两个字段:`trades.env` = 每笔的**事实**(demo/live,永远真值);`reconciliations.scope` = 本次对账的**口径**(all 现 / demo/live 未来)。回测侧不用标记 —— 它天然在 `backtests.trades`。

```
reconciliations
  strategy_id  scope('all' 现 / 'demo'|'live' 未来)  ┐ 主键 (strategy_id, scope)
  window_from  window_to        自动 = 实际成交时间跨度(§4)  │
  actual_trades  bt_trades      窗口内两边笔数               │
  match_score    综合匹配度 0~100(门禁/排名过滤用, 提列出来好查) │
  metrics       JSONB           所有对比细节 → 扩展全靠它     ┘
  updated_at
```

**`metrics` JSONB(同 `backtests.metrics` 套路,加维度不改表):**
```jsonc
metrics = {
  "count_match_rate": 0.94,    // 笔数一致率 0~1  = min(bt,actual)/max(bt,actual)
  "dir_match_rate":   0.92,    // 逐笔方向一致比例 0~1
  "result_match_rate":0.88,    // 结果(净盈亏)一致率 0~1
  "signal_hit_rate":  0.98,    // 信号命中率: demo/live有信号且回测也有 / 实际信号数
  // ↓ 未来加(调 PF/回测时), 表结构不动, 且届时按 env 分 demo/live:
  "pf_bt":1.4,"pf_actual":1.2,"pf_match_rate":0.86,
  "dd_bt":800,"dd_actual":950,"dd_match_rate":0.84
}
```

**指标统一口径:全部 `0~1`(展示 0~100%)、越高越好。** `match_score` = 上面加权 → 0~100。`match_score ≥ 90`(阈值进 config)= 回测质量达标,并列 OOS/健壮性进关2门禁。

**为何覆盖不存历史**:结论是"该策略回测当前可不可信"的**最新裁决**,随实盘累积要最新值,不要旧快照堆积。想看"校准后 match 有没有变好"→ Phase 3 再单独加 append-only `reconciliation_history`,不动本表。

## 4. 时间窗:系统自动,不用手填

```
window = [ min(实际成交 entry_time) , max(实际成交 exit_time) ]
```
- 回测那份(2015→今,已覆盖近期)**按窗口切片**,只取落在实际成交时段内的回测笔 → 和实际那几十笔比。
- **不用重跑回测**,现有 `backtests.trades` 直接切片。
- 只比**已平仓**回合(实际持仓中的不计入,待平仓后下次重算纳入)。

## 5. 维护模型(哪些更新 / 哪些覆盖 / 哪些未来维护)

| 对象 | 何时写 | 方式 | 长期维护 |
|---|---|---|---|
| `trades`(实际成交) | 心跳增量 | **upsert 只增**,永久 | 是(源真相,一直累积) |
| `backtests.trades` | 每次回测 | 覆盖 | 随回测 |
| `reconciliations` | **按需重算**(分析页/新成交后) | **覆盖**(策略×env 一行) | 重算即可,无历史包袱 |
| `reconciliation_history` | —— | append-only | **未来才建**(Phase 3,track 校准趋势) |

**重算触发**:现阶段**按需**(分析页点"重新对账");未来"有新成交自动标脏→重算"。
**扩展点**:新对比维度 → 进 `metrics` JSONB(不改表);分 demo/live → 加 env 行(不改表);历史趋势 → 加新表(不动现有)。
