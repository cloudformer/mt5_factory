# v5.0 worker 拓扑与策略挂载(整条 v5 的地基)

> 序列见 v5.1。本篇定死 worker 物理拓扑 + 策略如何挂到 worker——是多用户/市场/订阅的
> 共同前置,也可**独立于 v5 先上**(它是"当前一台 demo 机跑全部 demo 策略"这个扩容
> 天花板的正解)。2026-07-22 与 Frank 对齐:**直接上多挂载**(一策略可挂多台 worker),
> 单挂载只是它挂一行的特例——省得将来从单字段再迁一次。

## 一、拓扑:一台 Windows = 四位一体(1:1:1:1)

```
一台 Windows VM
   = 一个 bridge  (哑执行器, 用 MT5_PATH 独占终端连接)
   = 一个 runner  (策略循环, 等本机 bridge 连好再附着同一终端)
   = 一个 MT5 终端
   = 一个账户
   = mt5_hosts 表里的一行(= 一个 worker)
```

五者是同一实体的不同侧面,永远 1:1。**不在一台机器上塞多个终端**(MetaTrader5 包对一个
终端是进程级 IPC、非线程安全;两进程同时 initialize 会双双 IPC timeout——现状已按此设计)。
扩容 = 克隆一台 Windows = 多一行 worker。策略挂到 runner / windows / 终端是同一件事:
指到"哪台 worker"就唯一确定了 runner、终端、账户。

## 二、挂载 = strategy_mounts 表(一策略挂多 worker)

```sql
CREATE TABLE strategy_mounts (
    strategy_id INT REFERENCES strategies(id) ON DELETE CASCADE,
    host_id     INT REFERENCES mt5_hosts(id),   -- 挂到哪台 worker; env 由该 host 角色带出
    volume      DOUBLE PRECISION,               -- 每挂载点独立手数(空=用默认)
    enabled     BOOLEAN NOT NULL DEFAULT TRUE,
    PRIMARY KEY (strategy_id, host_id)          -- 一策略在一台 worker 上最多挂一次
);
```

- **env 不再是策略字段,由 host 角色带出**——挂到 demo 机=demo,挂到 live 机=live,
  防呆自动成立(不可能出现"env 与机器角色对不上");
- **runner 拉取极简**:`JOIN strategy_mounts m ON m.host_id = 我的host_id` ——
  每台 runner 用自己的 host_id 直接索引"指给我这台的策略集";
- **单挂载 = 挂载表里一行**,所以这套是单挂载的干净超集。

例:策略 100 三行挂载 = 同时在三个账户跑
```
(100, worker1)  worker1=demo → 跑 demo
(100, worker2)  worker2=demo → 跑 demo(另一账户/券商, 交叉验证)
(100, worker3)  worker3=live → 跑 live
```

## 三、为什么多账户天然安全(数据层已就绪, 不用动核心不变量)

- **trades 主键 = (account, position_id)**:同策略(同 magic=100000+id)在多账户的成交
  靠 account 分开,不撞;
- **magic 只需单账户内唯一**(runner `positions_get by magic` 只在本终端过滤)→
  多账户各有自己的 magic 100100,`magic=100000+id` 不变量**不动**;
- **账户唯一索引(schema/002)**:每个挂载点必是不同账户,杜绝"同账户双跑"。

## 四、必须跟着改的一处:strategy_stats 主键改按账户

现在 `strategy_stats` 主键 = `(strategy_id, env)`。策略挂两台 demo 机 → 两台都上报
`(100,DEMO)` → **互相覆盖**,成绩废掉。多挂载要求成绩按账户分:

```
主键改成 (strategy_id, account)   -- env 降为普通属性列
```

红利:对账从"回测 vs demo vs live"升级为**"回测 vs 每个账户并排"**,信息更全,
且正好落进已有视图体系(回测/实盘 × 汇总/明细/交叉),多一个账户=多一个可切换视图,
不是新引擎(遵守 v5.1 总纲:差异只在视图层)。

## 五、连带两处(顺带一次做全)

1. **手数搬到挂载点**:`strategies.volume`(每策略一个)→ `strategy_mounts.volume`
   (每账户可不同仓位),`strategies` 留默认;runner 每轮拉取即刷新,改了下一单生效;
2. **漏斗状态降为信息位**:策略可"一台 demo + 一台 live 同时挂",故 `status`
   (CANDIDATE/DEMO/LIVE)不再等于当前 env,退化为"走到过哪一关"的信息;
   真相是挂载表那几行。

## 六、生命周期(挂载驱动)

| 阶段 | 挂载 | 在哪跑 |
|---|---|---|
| CANDIDATE | 无挂载 | 只在库里回测, 不上任何 runner |
| 进 demo | 加一行挂到 demo 机 | 该 demo 账户 |
| 晋级 live | 再加一行挂到 live 机(可与 demo 并存) | demo+live 同时(持续对账) |
| 淘汰 | 清空挂载 | 自动停跑 |

## 七、与 v5 的衔接(为什么现在就上多挂载)

未来一策略会挂到很多地方,**每个 user 有自己的 worker,在自己 worker 间随便挂**——
多挂载正是这个的数据形状。owner 过滤叠在其上:`user → 他名下的 worker → 挂载行`。

- **自己的策略挂自己的 worker**:随便挂,参数是自己的,无限制;
- **订阅别人的 shared 策略(v5.7)= 给源策略加一行挂载**指向"跑订阅者账户的托管 worker"
  —— 不复制策略定义,每行带自己的 volume/enabled,比"影子实例"更优雅,
  **多挂载直接统一了"自己多账户跑"和"别人订阅跑"两件事**;
- **红线仍在**:订阅场景参数属于源策略作者,故那行挂载只能指向**平台托管 worker**
  (机器由平台控、只登订阅者账户),参数不落到订阅者自己机器的 runner 内存
  (自己的策略挂自己 worker 无此限制——参数本就是你的)。

## 风险 / 难度 / 验收

- 风险:**中**(strategy_stats 改主键是唯一 schema 硬改动;runner 认领键从"角色全量"
  改为"host 指派",靠挂载表空/回填平滑过渡);难度:中;
- 验收:策略挂多台 worker 各自出成交且账户分开不糊;strategy_stats 按账户并排;
  改某挂载点手数只影响该账户;清空挂载即停跑;单挂载(一行)行为与预期一致;
  runner 只拉指给自己 host 的策略。
