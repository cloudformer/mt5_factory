# v5.6 隔离与权限执法(第六步 · 唯一有变更风险的一步)

> 序列见 v5.1。前 5 步把字段/表/页面全部备好且零行为变化;本步一次接通:
> **认证 → 数据隔离 → 可见性放行 → 配额闸门 → 管理权限**。
> **铁则:本步验收全绿之前,绝不发任何用户 key;做完这步,系统才是多租户。**

## 内容(按接通顺序)

1. **认证**:api 全局中间件 —— `X-API-Key` → api_keys 查哈希 → 请求上下文带 user_id;
   web 登录(session);无凭证=401,白名单只放 /health;
   **worker 用独立 worker key**(mt5_hosts.worker_key_hash,权限只有拉任务/上报/心跳)
   —— 真金通道与用户 key 分离,泄露互不殃及;
2. **数据隔离**:所有查询按 `owner_id = 当前用户` 过滤 —— 收敛在一处依赖/中间件;
   runner 拉任务按 host.owner 过滤策略与配置;策略转 DEMO/LIVE 只认自己名下 worker;
3. **可见性放行**(按 v5.4 定死的语义):public 全可见+可 fork;shared 汇总可见+盲测;
   private 除 owner 外一律 404;**盲测=标准回测,引擎零分叉**——执法全在输出层:
   api 按可见性裁剪响应(shared 非 owner 只放行汇总视图,响应体断言无 trades),
   视图切换(回测/实盘 × 汇总/明细/交叉)变成权限问题而不是引擎问题;
4. **配额闸门**:usage_counters 表 + 四个入口(回测入队/AI报告/建策略/注册worker)
   查 quota_* 并原子计数(`ON CONFLICT DO UPDATE used=used+N`),超额 429 带人话;
   user_config 可写白名单:quota_* 只允许管理员写;
5. **管理权限**:v5.5 管理页只认超管。

```sql
-- 本步唯一新表
CREATE TABLE usage_counters (
    user_id INTEGER NOT NULL REFERENCES users(id),
    metric  VARCHAR(32) NOT NULL,
    period  DATE NOT NULL,                -- 按天一行, 跨天自然归零(新行)
    used    INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (user_id, metric, period)
);
```

## 可选加固(建议第二阶段再开): 数据库 RLS

应用层过滤跑通、越权扫描全绿后,再开 Postgres 行级安全兜底
(`CREATE POLICY ... USING (owner_id = current_setting('app.user_id')::int OR visibility IN ('public','shared'))`)
—— 代码哪天漏写 WHERE,数据库直接不给行;与"唯一约束防重复"同一哲学。
代价是调试与少量性能,所以分两阶段。

## 风险与缓解(本步 = 全路线的命门)

- **隔离漏一处=数据泄漏;worker 归属错一行=下错账户** —— 这是唯一真正危险的一步;
- 缓解三件套(全部纳入常驻回归清单,与 v2.4 测试哲学同源):
  ① 过滤收敛一处;② **越权扫描脚本**(两个测试用户互扫全部 API,断言零越权);
  ③ 无 key 扫全路由脚本(非 401 即 bug);
- 发布顺序:先 user 1 自己带 key 跑一周(行为应与今天逐字节一致)→ 建测试用户跑扫描
  → 全绿 → 才发第一把真实用户 key。

## 风险 / 难度 / 验收

- 风险:**高**(集中承担全部行为变更);难度:高;
- 验收:无 key 全 401;越权扫描全绿;user 1 全站回归无差异;
  配额打满被拒且提示明确;shared 盲测响应无逐笔;A 的 worker 拉不到 B 的任何东西。
