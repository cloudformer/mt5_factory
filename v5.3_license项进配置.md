# v5.3 license 控制项进配置(第三步 · 只加不变)

> 序列见 v5.1。原则(与 Frank 对齐):**凡是未来要被 license 控制的"量",现在都先做成
> 配置键**(config 全局默认 + user_config 可按用户覆盖)——未来 license/plan 直接关联
> 这些键,不用再改表。

## 功能

把配额型控制点种成 config 键(全局默认=你现在的实际用量习惯,宽松值):

```sql
INSERT INTO config (key, value) VALUES
    ('quota_backtests_per_day',  '100000'),   -- 每日回测次数上限
    ('quota_strategies_max',     '100000'),   -- 策略实例总数上限
    ('quota_workers_max',        '100'),      -- worker 数上限
    ('quota_ai_reports_per_day', '100000')    -- 每日AI成绩单/分析上限
ON CONFLICT (key) DO NOTHING;
-- 命名规约: quota_ 前缀 = 配额类, 与偏好类(volume_default等)一眼区分
```

**给某用户设额度 = 在 user_config 写他的覆盖行**(5.2 的机制原样复用,零新表):
```sql
INSERT INTO user_config VALUES (2, 'quota_backtests_per_day', '100');  -- 张三每天100次
```

**未来接 license/plan**:plan = 一组 quota 键值的模板,给用户套餐=批量 UPSERT 他的
user_config 覆盖行 —— 直接关联,表结构不用动。

## 本步不做(留给 5.6 及以后)

- 不做计数、不做拦截(那是行为变更):本步只是把"额度值"放进配置体系,谁都不读它;
- 用量计数表(usage_counters)与执法点(回测入队/AI报告/建策略/注册worker 四处)在
  5.6 之后按需上 —— 先有尺子,后装闸门。

## 一个必须记住的口径

**额度键用户自己不可覆盖**(不然自己改大=配额虚设)——user_config 的可写白名单里,
quota_* 只允许管理员写。白名单机制在 5.6 权限一起落,本步先写进文档立此为据。

## 风险 / 难度 / 验收

- 风险:**无**(种几个没人读的配置键);难度:低;
- 验收:键在库里、全局默认宽松到不影响你;user_config 能写能删覆盖行。
