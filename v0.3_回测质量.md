## 0. 先厘清:撮合是准的,漏的是"筛选"

`services/backtest.py` 的撮合已经足够悲观、足够真实,**不是问题所在**:

- 信号在 TF bar 收盘产生 → **下一根 M1 开盘价成交**(不偷看未来)
- 进场按 ask+点差+滑点,离场按 bid;**SL 先于 TP 检查**(同 bar 悲观)
- **跳空按实际开盘价成交**(sl_gap/tp_gap),不按挂单价
- 佣金按点数逐笔扣除;点差默认用每根 bar 记录的真实点差

所以"回测好 demo 差"**不是撮合失真**,而是:**没有反过拟合的筛选**——随机生成的一批参数里,总有几条在这段历史上碰巧漂亮;拿去 demo(=样本外)就现原形。

## 1. 反过拟合总方案(手段 / 治什么 / 代码改动风险 / 是否做)

**统一原理(所有手段共用一条)**:过拟合 = 把噪声当规律;而**规律可复制(换份数据还在),噪声不可复制(换份数据就没了)** → 检验永远是同一招:**在"没参与挑选"的数据上再考一次**,区别只是换哪个轴:

| 轴 | 手段 | 考题 |
|---|---|---|
| 时间轴 | #1 OOS 留出段 | 换个**时段**还赚吗? |
| 品种轴 | #5 跨货币对 | 换个**市场**还赚吗? |
| 券商轴(将来) | 跨券商数据 | 换个**报价源**还赚吗? |
| 参数轴 | #4 邻域稳健 | 参数**微调**还赚吗?(孤峰=拟合噪声缝, 平台=真结构) |

**demo/live 是终极样本外**(真·未来 + 真·执行);前面每道轴都是提前免费模拟这场终考。完整泛化链:
```
训练段选人 → 留出段(时间轴) → 跨货币对(品种轴) → demo(真未来) → live(真金)
```
双轴过滤要**一起用**(OOS=留出段盈利 + 健壮≥60%):单轴能骗(碰巧那个时段/那个品种), 同时骗过两轴的概率相乘、急剧变小。

| # | 手段 | 治什么 | 代码改动 / 风险 | 是否做 |
|---|------|--------|----------------|--------|
| — | **货币对 / 券商 筛选**(纯 filter) | 不是反过拟合,是**易用性**:圈定回测/查看哪些货币对、哪个券商的策略 | 只加两个筛选维度,不碰撮合、不跨品种 | ✅ **本期已实现** |
| **1** | **样本外切分**(时间维度) | 过拟合主因(demo 就是样本外) | 回测跑完按时间切 trades → metrics["oos"]={train, holdout};比例进 config(默认0.7=7:3, **长期固定, 换比例必全量收敛**);列表排名加"留出段"列 + "OOS=留出段盈利"一票否决过滤。**纯后处理,撮合零改动**。合成数据验证:切分点精确、笔数/净点守恒 | ✅ **已实现**(2026-07-14) |
| **2** | **分段最少笔数** | 小样本假象 | 训练段、留出段各自都要够笔数;`/backtest/top` 筛选加强。极低风险 | 🔜 跟 #1 一起 |
| **3** | **短样本警告** | 数据太浅放大一切过拟合(现仅~3个月) | 回测算 M1 时间跨度,不足门槛标"样本不足,结论不可靠"。低风险 | 🔜 跟 #1 一起 |
| **5** | **跨品种健壮性**(空间维度) | 品种特异性过拟合(只在 XAU 历史巧合赚) | 勾『跨品种验证』每策略在所有货币对各跑一行;排名加"N/M 赚"健壮性列 + 点开每品种明细。存储键 (strategy_id, symbol) upsert | ✅ **已实现**(与 #1 正交) |
| **4** | **参数邻域稳健性** | 参数卡在历史噪声缝里(孤峰) | 对候选生成 ±10~20% 邻居批量回测,要求邻居也普遍赚。新编排,算量 ×N | 🔶 第二期 |
| **6** | **walk-forward** | OOS 的滚动升级(金标准) | 多窗口编排,大改 | ❌ 暂不做 |

> **注(本期取舍):** 一度实现过 #5(跨品种健壮性列),后与用户确认**当前只要纯 filter(甲)**、不要跨品种展开(乙),故 #5 已回退为规划。#5 与本期的"货币对筛选"不是一回事:筛选是"每个策略只跑自己的货币对,选看哪些";#5 是"同一参数在别的货币对上也跑,看是否普适"。

**数据深度不改代码,却是地基**:引擎不关心 3 个月还是 10 年,但样本越深、OOS 越可信。深度是"下载/存储"的事,与回测逻辑无关。

## 2. 本期实现:货币对 + 券商 两个筛选(纯 filter)

回测**按货币对进行**(每个策略跑它自己的货币对,一如既往)。本期在回测页加**两个筛选维度**,决定"回测/查看哪些策略":

```
批量回测  [状态] [上限] [策略ID]  货币对[全部▾]  券商[全部▾]  [成本…]  [启动]
结果排名  货币对[全部▾]  券商[全部▾]  [最少交易数]  [筛选]
```

- **货币对**:全部 = 不限;选具体货币对 = 只回测/只看该货币对的策略。
- **券商**:全部 = 不限;选具体券商 = 只回测/只看该券商的货币对(按 `symbols` 表的券商标签圈定品种)。
- 选定券商 + 货币对全部 = **该券商的所有货币对**。两个维度全按选项走。
- 下拉选项从**库里** group by 出来(货币对 = download 品种;券商 = 品种主档里出现过的券商标签),默认"全部"。

### 券商这个标签从哪来(关键,别再混)
- **只有"下载数据"跟 worker 打交道**:下载时把**当前券商**(worker 账户 server 名)写进 `symbols.broker`。这是券商唯一的落库时刻。
- **回测之后完全不碰 worker**:回测只读数据库,按库里已有的券商标签过滤,和 worker/在线状态无关。
- **原来**没有券商列 → 等价于"当前货币对 × 所有券商"(其实就一家、没打标、没这概念)。现在券商成了可选筛选维度。
- 今天只有一家券商 → 下拉就一个选项,选"全部"照跑;接了第二家自动多出来。
- 补充:同一货币对"多家券商价格序列并排比"是另一回事,需**重量版 schema**(broker 进 `historical_bars` 主键),与本期筛选无关。

## 5. 实测发现(2026-07-12 首次全量回测, 550 策略)

| 观察 | 数据 | 说明 |
|------|------|------|
| **upsert 生效** | 回测行数 550 = 涉及策略数 550 | 一策略一行, 表不增长 ✓ |
| **净点"对称"= 过拟合信号** | 各币种 最高净点 ≈ \|最低净点\|:XAU +43449/−40061、EUR +6608/−7070、GBP +7383/−6759、JPY +7483/−9553 | 随机策略在噪声上的典型长相(有人蒙对赚大、有人赔大, 近似抛硬币)→ **直接印证:单靠全样本净点排名极易挑到运气选手** |
| **样本薄** | EUR/GBP/JPY 平均 13–19 笔, XAU 78 笔 | 多数不到 30 笔 → 被排名 `min_trades` 过滤后所剩无几, 结论也不稳(数据仅~3个月) |
| **全样本最优也普通** | XAU 冠军:258笔 / 胜率44% / PF 1.22 / net +43449 / 最大回撤 14431 | PF 1.22 普通, 靠盈亏比撑正期望;**未经 OOS 不可信** |
| **孤儿** | BTCUSD 150 条(券商空) | 品种已删的孤儿策略, 由回测页警告条归档 |

**结论:实测直接印证了过拟合风险 —— 全样本净点排名会挑到运气选手。#1 OOS 时间切分是当务之急。**

## 6. 跨品种健壮性三级显示(已实现)

结果排名的三级颗粒度,主行不炸 + 一眼看健壮 + 点开看细分:

```
主行: ID │ 名称 │ 主品种 │ …回测指标… │ 跨品种健壮性 ▾4/5赚🟢 │ 状态 │ 操作
  点 ▾ 展开(复用 worker 的 subtable): 每品种一行 品种│券商│笔(胜/负)│净点│PF│回撤
```

| 层 | 显示什么 | 回答什么 | 颗粒度 |
|----|---------|---------|--------|
| 主行·回测列 | 主品种成绩 | 真去 demo/live 会怎样 | 粗 |
| 主行·健壮性列 | 跨品种:`N/M 赚`(全赚🟢/≥60%🟡/少数⚪);单品种:`1 品种`。**都可点开** | 普适 还是 巧合 | 摘要 |
| 展开明细 | 每品种/券商一行(主品种带`主`标),**按当前 DB 实际有的行展示,不强求多维度** | 适配哪个品类 / 哪个不行 | 细 |

**展开随时可点**:哪怕只跑了主品种(1 行),也能点开看那一行(按当前 DB 展示);勾『跨品种验证』后才会有多品种行、健壮性列才显示 `N/M 赚`。
**排名纪律**:排名/晋级只认**主品种行**(`b.symbol = s.symbol`),跨品种结果只喂健壮性,不参与排名。

## 7. 待查:回测预览数 与 排名列表数 对不上(2026-07-14 观察, 未定位)

**现象**:回测页"将回测 3600 个策略",排名页"复制ID"却 12900 个,差 ~9300。

**已知/待核实**:
- 排名 `top()` 默认**含所有状态**(含淘汰归档 ARCHIVED);回测 `run/plan` 经 GAP A 改动后默认**排除 ARCHIVED** → 两页"全部"口径不一致,是差额的一部分。
- 淘汰归档实测仅约 300(不足以解释 9300)。
- 回测 `plan` 还要求"品种仍在主档"(`symbol IN symbols`),**孤儿**(品种已删/未登记)排名有、回测无 → 疑似主因,未核实数量。
- 单批上限 `backtest_batch_limit` 若很小会把 plan 的数**截断**,则"3600"非真实可测数 — 未确认其配置值。
- "复制ID"曾粘出 `ID,券商` 成对(应为纯 ID 串)→ 疑似复制逻辑 bug 或手动选列,待复现。

**定位手段(只读 SQL, `make psql`)**:
```sql
SELECT status, count(*) FROM strategies GROUP BY status;                                    -- 各状态分布
SELECT count(*) FROM strategies WHERE symbol NOT IN (SELECT symbol FROM symbols);            -- 孤儿数
SELECT value FROM config WHERE key='backtest_batch_limit';                                   -- 单批上限
SELECT symbol, count(*) FROM strategies GROUP BY symbol ORDER BY count(*) DESC;              -- 品种分布
```

**待办**:①统一两页"全部"口径(建议排名页默认也隐藏淘汰归档,要看再选);②核实孤儿数并在排名页也给孤儿提示/过滤;③复现并修"复制ID"串列问题。

## 7.5 回测质量的第三维:关2 对账匹配度(见 v0.6)

回测质量不止"反过拟合"(OOS/邻域/健壮),还有一维是**"回测信/实盘验"**:回测信号/结果与 demo/live 实际成交对不对得上 → 综合成 `match_score`(0~100)。这部分设计与落地在 **`v0.6_关2对账.md`**,展示落在**分析▾→回测质量分析**页,`match_score≥阈值` 与 OOS/健壮性并列进关2门禁。

## 8. 本期做:排名页分页加载(性能)

排名页一次性加载全部策略(远程 ~12900 行 ≈ 6MB 载荷 + 约 2.6 万 DOM 行,叠加 ngrok 隧道 → 明显卡)。改为**服务端分页**:每页只返回一页(默认 100),跨品种明细**只为当页拉取**(而非全部),大幅降载荷与 DOM。全局排序/筛选仍在完整集合上算后再切页,保持排名语义不变。
