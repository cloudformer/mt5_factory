# v1.2 数据持久化(成交回写数据库)

> 状态:**规划中,先做**(优先级高于策略功能扩展)。
> 目标:把成交明细从"每次现拉 MT5"改为"持久化进本地库",消除多源打架、摆脱 90 天/worker 在线依赖。

---

## 1. 要解决的隐患:战绩现有 3 个独立数据源,会打架

| 数据源 | 喂给哪个页面 | 特性 |
|--------|-------------|------|
| `strategy_stats` 表(心跳快照) | Demo/Live 页"实跑"列 | 30 秒一刷的快照,滞后 |
| `/recon`(独立现拉 MT5) | 对账页 | 实时现拉 |
| `/trades`(独立现拉 MT5) | MT5 流水页 | 实时现拉 |

**同一批成交,三条独立代码路径各算各的** → 快照滞后于现拉、90 天窗口口径不同 → Demo 战绩与 MT5 流水短暂对不上。**没有唯一真相源。**

另外:`HISTORY_DAYS=90`(conn/stats.py)是我们自己定的,不是券商限制——超 90 天的老单滑出窗口,战绩数字会变小。

---

## 2. 目标:一个源,汇总与明细数学上永远一致

```
trades 表(每笔成交腿, 主键 (account, ticket))  ← 唯一真相源
   ├─ 战绩(Demo/Live 实跑列) = 对 trades 按 magic 聚合(SUM)
   └─ 历史成交明细(MT5 流水"历史成交") = trades 的原始行
```

关键:**战绩是明细的 SUM,两者是同一张表的两种视图 → 不可能不一致。**

**分界(重要):**
- **已发生的(成交/平仓)** → 回写 DB,统一、一致、不限 90 天
- **进行中的(当前持仓/浮动盈亏)** → 永远实时拉 MT5(活状态,还没成为历史,不存库)

---

## 3. 一致性四道锁(回写前必须做到)

1. **deal ticket 唯一键 + `ON CONFLICT DO NOTHING`** — 每笔成交有 MT5 全局唯一 ticket;主键 (account, ticket) → 重复同步 100 次也只有一行,永不重复计
2. **重叠窗口同步** — 每次不只同步"上次之后",而是往前多覆盖一点(如多拉 1 天),防边界漏单;反正 ON CONFLICT 去重,多拉无害
3. **以券商为准** — DB 是 MT5 的持久化副本,任何不一致下次心跳重拉那段 upsert 自动补齐;**券商永远是唯一真相源,DB 是从**
4. **交叉验证** — DB 算的 vs MT5 现拉的定期比对(同现在自检的 cross-check),对不上告警 → 用"两条独立路径必须一致"守正确性

---

## 4. 无缝衔接:现拉降级为"补数据",不推倒重来

```
现在:  战绩 = 每次去 MT5 现拉近 90 天算
v1.2:  成交进 trades 表 → 战绩 = 查 DB 累计(不限 90 天)
       worker 心跳仍连 MT5, 但只做"把新成交增量同步进 DB"
```
- 第一次全量把 MT5 能给的历史灌进 DB;之后每次心跳只同步 DB 里还没有的新成交
- 查询从"现拉 MT5"切到"读 DB",用户无感,只是数字不再受 90 天限、不受 worker 在线限
- 统一查询源:Demo 战绩 / 对账 / MT5 流水"历史成交" 全部读 trades 表,不再各拉各的

---

## 5. 落地映射(草案)

| 层 | 改动 |
|----|------|
| schema | 新增 `trades` 表:account, ticket(PK), position_id, magic, strategy_id, symbol, type, entry(in/out), reason(sl/tp/…), volume, price, profit, commission, swap, time(券商时间) |
| worker/bridge | 已有 `/trades`(现拉),不变;新增"增量同步"由 api 心跳驱动 |
| api | 心跳时把每个已连接账户的新成交 upsert 进 trades(重叠窗口);战绩查询改为对 trades 聚合;`strategy_stats` 降级为缓存或从 trades 重算 |
| web | Demo/Live 战绩、对账、MT5 流水"历史成交" 统一读 trades;"当前持仓"仍实时拉 |

---

## 6. 顺带解决

- **孤儿/跨券商**:trades 带 account,天然按账户隔离;将来多券商也不混
- **对账自动化(关 2)**:成交进库后,"回测预期 vs 实际成交"逐笔对比可直接在库里做,不再依赖 worker 在线
- **永久战绩**:不受 90 天窗口限,策略长期表现可完整回溯

---

## 7. demo/live 逐笔明细(细化:UI 放策略后面 + 逐笔含入/出时间)

**分层原则(和回测对照):**
| | 存法 | 为什么 |
|---|------|--------|
| 回测 | 每(策略×品种)**一行汇总**(已实现) | 候选量大,逐笔存不起 |
| **demo/live** | **每笔一条**(含入/出时间) | demo/live 策略**数量少** + bar 级低频 → 逐笔存得起;且**只有逐笔才能对账 + 精细评价** |

**粒度:按"回合(round-trip / position)"存,一次完整开平=一条**(细化 §2/§5 的"deal 腿" → 更贴"每笔一条、含入出时间"):
```
account · position_id(PK, 券商唯一) · magic→strategy_id · demo/live · symbol · 方向
入场时间 · 入场价 · 出场时间 · 出场价 · 平仓原因(sl/tp/手动)
profit · commission · swap(券商真实数)
```
- MT5 原始是 deal 腿(开一腿、平一腿);同步时按 `position_id` 配对成**一条回合**。四道锁(§3)不变,ON CONFLICT 键换成 `(account, position_id)`。
- 部分平仓(一个 position 多次平)= 多条回合行,按平仓腿拆;bar 级策略通常一次全平,少见。

**UI(你要的"放到策略后面"):**
- Demo/Live 页**策略行 = 战绩汇总**(笔数 / 胜负 / 净盈亏 = 对该策略 trades 求和)。
- **点开展开逐笔**(入场时间 → 出场时间、盈亏、sl 还是 tp)—— 复用回测结果页那套"展开明细"组件。

**数据量 / 维护:** 少策略 × 低频 → 一年几百笔/策略、几千行封顶,忽略不计;心跳**增量 upsert**,重复同步无害。

---

## 8. 策略评价:如何加权(P2, 阈值进 config)

不是简单 `sum(profit)`。按重要性排:
1. **对账偏离(能不能信,最高优先)**:demo 逐笔 vs 回测预期逐笔比,偏离大 = 回测对它不准 → 先降级,盈亏都不用看。
2. **live > demo**:live 真金是终审,demo 只是彩排;有 live 以 live 为准。
3. **时间衰减**:近期交易权重高(市场 regime 会变,老交易参考价值低)。
4. **样本量置信**:笔数不够不下结论(同回测 min_trades)。
5. **风险调整**:看 净利/最大回撤、连亏次数、单笔方差,不只看总净利。

→ 综合成"实盘评分"或直接做 **DEMO→LIVE 准入门禁**(对账达标 + 样本够 + 风险可控才放 live)。所有权重/阈值进 config,不写死。

- **P2**:策略评价加权 / DEMO→LIVE 门禁(§8)。
