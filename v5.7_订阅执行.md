# v5.7 订阅执行(市场变现闭环 · 依赖 5.6 先立住)

> 序列见 v5.1。商业模式(2026-07-22 与 Frank 对齐):**平台方自己是首个供给方**——
> Frank 开发的好策略打成 shared 挂到市场,真实实盘战绩吸引用户。
> 完整闭环:**看汇总(免费)→ 盲测验证(占配额)→ 订阅执行(付费)**。

## 语义:shared 的"用"

shared 策略别人拿不到参数(红线),所以"用"只能是**订阅执行**:
订阅者出账户,系统替他跑这份策略——参数只在服务端流转,永远不落到订阅者手里;
手数/启停订阅者自己控制,盈亏归他,平台按订阅收费。

## 数据:订阅 = 给源策略加一行挂载(复用 v5.0 多挂载, 不建影子实例)

订阅**不复制策略**,就是在 v5.0 的 `strategy_mounts` 里加一行:

```
(strategy_id=源策略, host_id=跑订阅者账户的托管worker, volume=订阅者设, enabled=订阅者控)
```

- 源策略定义只有一份(owner=作者),挂载行把它下发到订阅者账户的 runner;
- magic=100000+源id 在订阅者账户内唯一,trades 靠 (account, position_id) 天然分开,
  归因/对账照 v5.0 走(回测 vs 该账户并排),**不用 is_subscription、不用影子实例**;
- 源策略被 owner 停/删 → 该挂载行随之失效(订阅者收到明确状态,不悄悄失效);
- 手数/启停 = 订阅者改自己那行挂载的 volume/enabled;
- 配额新维度:`quota_subscriptions_max`(可订阅几个,进 v5.3 的 quota_* 体系)——
  即"你能拥有几行指向别人源策略的挂载"。

## 关键约束:那行挂载只能指向平台托管 worker

参数不给订阅者看,但策略要跑就得进 runner 进程内存——
**若挂载指向订阅者自己的机器,参数就能被内存抓走,红线形同虚设**。
所以订阅挂载定死:**host 只能是平台托管的 worker**(机器平台控、只登订阅者账户):

- 托管 worker 的 mt5_hosts.owner=平台(user 1),但登录的是订阅者账户;
- 与"自己的策略挂自己 worker"区分:自己策略参数本就是你的,随便挂;
  订阅是别人的参数,故限定托管 worker——api 在建挂载时按"源策略 owner ≠ 挂载目标 owner"
  这一条判定为订阅,强制目标为平台托管 host;
- 账户唯一索引(002)照旧防双跑;runner 链路零改动(它只认 host_id → 策略集 → 账户);
- 自然衔接托管费/workers_max——"自己 worker 跑自己策略免费,托管跑订阅另算"。

## 回测与视图(与 v5.4 同一口径)

订阅前的验证就是盲测=标准回测(引擎零分叉,逐笔照常入库);
订阅后的实盘明细是**他自己账户的成交**,对他全可见(那是他的钱),
但回测明细/交叉比对里涉及参数反推的视图仍按 shared 红线只给汇总。

## 风险 / 难度 / 前置

- 前置:**5.6 验收全绿**(参数保护/隔离/配额闸门全部立住,本步全靠它们);
- 风险:**中-高**(真金代跑,源策略变更联动、托管 worker 容量、收费对账都是运营题);
- 难度:中(数据与执法机制全是复用,新东西主要是订阅关系管理+托管运营);
- 验收:订阅者加一行挂载→托管 worker 下单→trades 落他账户→他看不到 params 任何途径;
  源策略停用→挂载行失效+状态可见;退订=删该挂载行;订阅挂载被强制限定平台托管 host。
