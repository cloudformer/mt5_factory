# v5.7 订阅执行(市场变现闭环 · 依赖 5.6 先立住)

> 序列见 v5.1。商业模式(2026-07-22 与 Frank 对齐):**平台方自己是首个供给方**——
> Frank 开发的好策略打成 shared 挂到市场,真实实盘战绩吸引用户。
> 完整闭环:**看汇总(免费)→ 盲测验证(占配额)→ 订阅执行(付费)**。

## 语义:shared 的"用"

shared 策略别人拿不到参数(红线),所以"用"只能是**订阅执行**:
订阅者出账户,系统替他跑这份策略——参数只在服务端流转,永远不落到订阅者手里;
手数/启停订阅者自己控制,盈亏归他,平台按订阅收费。

## 数据(影子实例, 复用现有体系)

订阅 = 在订阅者名下建一个**影子实例**:

```sql
-- strategies 复用 v5.4 的 origin_id 指回源策略, 再加一个标记
ALTER TABLE strategies ADD COLUMN is_subscription BOOLEAN NOT NULL DEFAULT FALSE;
-- 影子实例: owner=订阅者, origin_id=源策略, params 从源实时引用(源调参订阅者跟随),
-- 对订阅者 params 永不可见(输出层裁剪, 同盲测红线); 独立 id → 独立 magic → 归因照旧
```

- 影子实例独立走自己的 magic number,trades/对账/统计全部照现有体系落在订阅者名下;
- 源策略被 owner 停/删 → 影子实例联动停(订阅者收到明确状态,不是悄悄失效);
- 配额新维度:`quota_subscriptions_max`(可订阅几个策略,进 v5.3 的 quota_* 体系)。

## 关键约束:订阅执行必须跑在平台托管 worker 上

参数不给订阅者看,但策略要跑就得进 runner 进程内存——
**worker 若是订阅者自己的机器,参数就能被内存抓走,红线形同虚设**。
所以订阅执行定死:**只跑在平台托管的 worker 上**(订阅者只出 MT5 账户,机器平台管):

- mt5_hosts 已有 owner_id(v5.2):托管 worker 归平台(user 1),挂订阅者的账户跑影子实例;
- 账户唯一索引(002)照旧防双跑;runner 链路零改动(它只认策略+账户,不关心订阅关系);
- 自然衔接托管费/workers_max 配额——"自己的 worker 跑自己的策略免费,托管跑订阅另算"。

## 回测与视图(与 v5.4 同一口径)

订阅前的验证就是盲测=标准回测(引擎零分叉,逐笔照常入库);
订阅后的实盘明细是**他自己账户的成交**,对他全可见(那是他的钱),
但回测明细/交叉比对里涉及参数反推的视图仍按 shared 红线只给汇总。

## 风险 / 难度 / 前置

- 前置:**5.6 验收全绿**(参数保护/隔离/配额闸门全部立住,本步全靠它们);
- 风险:**中-高**(真金代跑,源策略变更联动、托管 worker 容量、收费对账都是运营题);
- 难度:中(数据与执法机制全是复用,新东西主要是订阅关系管理+托管运营);
- 验收:订阅者建影子实例→托管 worker 下单→trades 落他名下→他看不到 params 任何途径;
  源策略停用→影子联动停+状态可见;退订=影子实例停用归档。
