## 总纲:不分叉逻辑,差异只在三层(2026-07-22 定,全 v5 受此约束)

多用户不引入任何业务逻辑分支——**引擎/回测/撮合/对账/runner 全系统只有一条代码路径,
对所有用户逐字节一致**。用户之间的差异只体现在三个"薄层",且都是加法:

1. **UI 视图层**:同一份数据,按权限决定放行哪个视图(回测汇总/明细、实盘汇总/明细、交叉比对);
   "盲测""市场""订阅"都不是新引擎,是老数据的不同视图 + 不同放行规则;
2. **权限层**:owner_id 过滤 + 可见性放行 + 认证,统一收敛在一处中间件,不散落进业务代码;
3. **配额层**:quota_* 配置 + 用量计数,在动作入口"查+计",不改动作本身的逻辑。

好处:代码只有一份好维护、好审查、好控制;新用户=数据加行,不碰代码;
任何一步都可停可回退。**违反此纲(为某用户/某可见性写 if 分支进引擎)= 欠账,先记再还。**

## 功能

strategies 加两个字段:owner_id(归属人)、visibility(可见性),存量全归用户1(=你);
页面给一个**改 visibility 的入口**(策略列表操作列,下拉:private/public/shared,
和手数下拉同款交互)。只是打标,不产生任何可见性行为——真正按可见性放行/隐藏在 5.6。

```sql
    id SERIAL PRIMARY KEY,
    name VARCHAR(64) NOT NULL UNIQUE,

    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
INSERT INTO users (id, name) VALUES (1, 'frank') ON CONFLICT DO NOTHING;

ALTER TABLE strategies ADD COLUMN owner_id INTEGER NOT NULL DEFAULT 1 REFERENCES users(id);
ALTER TABLE strategies ADD COLUMN visibility VARCHAR(8) NOT NULL DEFAULT 'private'
    CHECK (visibility IN ('private', 'public', 'shared'));
```
